# 6단계 워크플로우

## Overview

이벤트 스토밍은 **알베르토 브란돌리니 (Alberto Brandolini, 2013)** 가 제안한 도메인 분석 워크숍 기법입니다. 이 도구는 그 표준 절차를 디지털화하여, 한 화면에서 6단계를 순차적으로 따라갈 수 있게 합니다.

상단 STEP 바에서 단계를 전환하면 해당 단계까지의 포스트잇만 보이고, 후속 단계 카드는 자동 숨김됩니다.

---

## STEP 1 · 도메인 이벤트 도출

> 🟧 **오렌지 포스트잇** · `#F5A623`

### 핵심 질문
"이 시스템에서 어떤 일이 일어나는가?" — **과거형 사실** 만 채집.

### 명명 규칙
- 항상 과거형, `~됨` 종결 (`주문생성됨`, `결제승인완료됨`, `재고차감됨`)
- `~되었다`, `~한다`, `~할 것이다` 금지 (현재형은 커맨드)
- 비즈니스 의미가 중요 — 기술적 액션 (DB INSERT 등) 은 이벤트 아님

### 예시
- ✅ `주문생성됨`, `결제승인완료됨`, `재고차감됨`, `배송출고됨`
- ❌ `주문하기`, `결제 처리 중`, `재고를 차감했다`

### 팁
- 가장 비즈니스 가치 높은 이벤트부터
- 한 도메인당 5-15 개 핵심 이벤트
- 너무 미시 이벤트 (CRUD 한 칸씩) 은 통합

---

## STEP 2 · ＋ 액터 식별

> 🟨 **노랑 포스트잇** · `#FFE066`

### 핵심 질문
"누가/무엇이 이 이벤트를 발생시키는가?"

### 액터 분류
- **사람**: 고객·관리자·판매자·CS상담사·관제요원
- **내부 시스템**: 정산배치·추천엔진·검색엔진·시스템(자동 트리거)
- **외부**: PG사·택배사·SMS사·이메일사

### 마스터 데이터 입력
액터별로 다음 메타 추가:
- 액터명·유형·분류
- 액터 설명 (한 줄)
- 주요 역할 (어떤 책임을 갖는가)
- 비고 (보안/감사 요건 등)

> ➡️ PPT 출력 시 액터 카드, 헥사고날 좌측 액터 패널, RBAC 매트릭스에 활용됩니다.

---

## STEP 3 · ＋ 커맨드 도출

> 🟦 **민트/시안 포스트잇** · `#6BCFE5`

### 핵심 질문
"어떤 명령이 이 이벤트를 트리거하는가?"

### 명명 규칙
- 현재형 동사 (`주문생성`, `결제승인`, `재고차감`)
- 액터의 의도를 표현
- 이벤트와 1:1 또는 N:1 관계

### 예시
| 액터 | 커맨드 | → 이벤트 |
|------|--------|----------|
| 고객 | 주문생성 | 주문생성됨 |
| 시스템 | 결제승인 | 결제승인완료됨 |
| 정산배치 | 회원등급재산정 | 회원등급변경됨 |

### 팁
- 커맨드는 "사용자가 누른 버튼"·"시스템이 호출한 메서드" 라고 생각
- ApplicationService 메서드명 (PlaceOrder·ApprovePayment) 과 1:1 매핑

---

## STEP 4 · ＋ 애그리거트

> 🟩 **연두 포스트잇** · `#A0E050`

### 핵심 질문
"어떤 도메인 객체가 일관성을 유지하는가?"

### DDD 애그리거트 원칙
1. **하나의 트랜잭션 = 하나의 애그리거트** — 한 트랜잭션에 여러 애그리거트 변경 ❌
2. **작게 유지** — 큰 애그리거트는 락 경쟁·메모리 부담
3. **외부 참조는 ID 만** — `order.customer` ❌ → `order.customerId` ✅
4. **Aggregate Root 가 모든 상태 변경 메서드 통제**

### 명명
- 영문 PascalCase
- 도메인 객체명 (도구/패턴이 아님)
- 예: `Order`, `OrderLine`, `Payment`, `Inventory`, `Subscription`, `BillingCycle`

### 다중값
한 이벤트가 여러 애그리거트 영향 가능 — 쉼표로 구분 (`Order, OrderLine`)

---

## STEP 5 · ＋ 외부 / 정책 / 핫스팟

### 외부시스템 🟪 핑크 · `#FFB6C1`

"어떤 외부 시스템과 상호작용하는가?"

마스터 메타:
- 시스템명·**포트명** (헥사고날 outbound port 인터페이스)
- 유형 (REST/Kafka/MQ/DB/...) · 프로토콜 · 인증
- 연계방향 (Inbound/Outbound/Both)
- 운영주체 · SLA (Critical/High/Medium/Low)
- 설명

### 정책 (Policy) 🟪 보라 · `#C9B6E8`

`when X then Y` 형태의 자동 트리거 규칙:
- "재고/가격 검증 + Saga 시작"
- "5회 실패 시 30분 잠금"
- "D+7 자동 구매확정"
- "30일 휴면 후 PII 마스킹"

### 핫스팟 (Hotspot) 🟥 핫핑크 · `#FF1493`

워크숍 중 발견된 미해결 의문·리스크·결정 보류:
- `결제실패`, `재고부족`, `위변조시도`, `선착순폭주`, `카드정보유출`, `비밀번호공격`

> 핫스팟은 후속 미팅에서 다루기로 표시한 메모이며, PPT 출력 시 회전된 핑크 라벨로 강조됩니다.

## STEP 6 · ＋ 바운디드 컨텍스트

> 점선 박스로 BC 그룹화

### 식별 신호
1. **용어 충돌** — 같은 단어가 다른 의미
2. **다른 stakeholder** — 다른 부서 책임
3. **다른 변경 주기**
4. **다른 SLA**
5. **다른 데이터 모델**

### Core / Supporting / Generic 분류

- **Core** — 경쟁 우위의 핵심. 직접 개발 + 가장 좋은 인력 (예: 추천 엔진, 동적 가격)
- **Supporting** — 핵심 보조, 차별화 아님. 직접 개발 가능 (예: 권한 관리, 알림)
- **Generic** — 어디나 똑같음. SaaS/오픈소스 (예: OAuth, PG, 이메일)

### 1 BC = 1 마이크로서비스 = 1 DB

DDD/MSA 표준 매핑:
- 1 BC → 1 헥사고날 → 1 마이크로서비스 (배포 단위)
- 1 BC → 1 데이터베이스 인스턴스 (다른 BC 가 같은 DB 직접 접근 ❌)

### Context Mapping

BC 간 통합 패턴 (어떤 관계로 협업하는가):
- **Customer/Supplier**
- **Conformist** (수용형)
- **Anti-Corruption Layer (ACL)** — 외부 레거시·SaaS 통합 시 권장
- **Open Host Service / Published Language** — 표준 API 제공
- **Shared Kernel** — 공유 라이브러리
- **Separate Ways** — 통합 안 함

## 워크플로우 결과물

6 단계가 모두 채워지면:

📊 **엑셀** → 🧩 **스토밍 편집** → 📑 **7종 산출물 (XLSX·PPTX)** → 🏛 **MSA 청사진**

특히 **헥사고날 다이어그램 (BC × 3 슬라이드)** 가 풀로 그려져, 그대로 개발팀에 전달 가능한 청사진이 됩니다.
